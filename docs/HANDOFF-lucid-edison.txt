== WHAT WAS BUILT (3 rounds, all on the writing surface) ==

1) IMAGE ATTACHMENTS — attach images (upload, or "take a photo" on mobile via
   <input capture>) to a page, from BOTH the Today composer and the entry view.
   - Bytes are app-encrypted (AES-256-GCM, the SAME ENCRYPTION_KEY as journal
     text) and stored in Replit Object Storage. Postgres holds only metadata +
     the storage key. Images are NEVER sent to the memory engine and NEVER
     resurfaced — they appear only when the writer opens the full page.
   - New table: entry_attachments. New endpoints (owner-scoped):
       POST   /api/entries/:id/attachments     (raw image via express.raw)
       GET    /api/entries/:id/attachments      (metadata list)
       GET    /api/attachments/:id              (streams DECRYPTED bytes)
       DELETE /api/attachments/:id
   - Account delete now purges image objects from the bucket; /privacy/export
     JSON gained an `attachments` array.

2) RICH TEXT — a literary formatting layer (bold, italic, H2/H3, bullet+numbered
   lists, blockquote, a few MUTED on-brand text colors) in both editors.
   - ENGINE-SAFE BY CONSTRUCTION: `body` stays the canonical PLAIN TEXT the
     engine, resurfacing excerpts, Library previews, and exports read. A new
     nullable, encrypted `body_rich` column holds sanitized HTML (display only).
   - The SERVER is the trust boundary: when a request carries `bodyRich`, the
     server sanitizes it (sanitize-html allowlist) and DERIVES `body` plaintext
     from the sanitized HTML. Client markup is never trusted into `body`; stored
     HTML is always safe to render. Entries without formatting render as before.

3) VOICE DICTATION — a mic in the editor toolbar (so it's in both editors).
   Uses the browser Web Speech API: live interim caption, finalized transcripts
   inserted at the caret, honest in-UI disclosure that the browser's speech
   service handles audio and we don't record it. Feature-detected (hidden in
   Firefox). Frontend-only — no API/schema/dep changes.
